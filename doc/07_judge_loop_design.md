# LLM AS A JUDGE - ループ設計ドキュメント

## 概要

「LLM AS A JUDGE」パターンは、別のLLMを評価者として用いて、生成された小説の品質を客観的に評価し、改善ループを自動化するアプローチです。

**重要な転換**: 従来の人間基準の評価軸から、**AI視点での評価軸**へシフト。AIが「面白い」「興味深い」と判定する基準を優先します。

## アーキテクチャ

```
┌─────────────────────────────────────────────────────────┐
│                  07_judge_llm.py                        │
│  LLM AS A JUDGE パターンの実装                          │
└─────────────────────────────────────────────────────────┘
              ↑ 入力                     ↓ 出力
              │                         │
     ┌────────────────┐        ┌────────────────────┐
     │ 06_generate    │        │ judge_results/     │
     │ _wagahai.py    │        │ (評価ログ JSON)    │
     │ の出力ファイル  │        │                   │
     └────────────────┘        └────────────────────┘
```

## ループの流れ

### フェーズ1: 初期評価 (Round 1)

```
初期生成ファイル
     ↓
┌─────────────────────────────────────┐
│  LLMJudge.evaluate_story()          │
│  ・プロット整合性 (0-10)            │
│  ・文体品質 (0-10)                  │
│  ・新奇性 (0-10)                    │
│  ・感情的インパクト (0-10)          │
│  ・テーマ表現 (0-10)                │
│  ↓ 計算                             │
│  総合評価 = (各項目の平均)           │
└─────────────────────────────────────┘
     ↓
  スコア 出力
```

### フェーズ2: 判定と分岐

```
総合スコア >= 閾値 (デフォルト 7.5)
     ↙ YES           ↘ NO
   終了          改善提案生成
                     ↓
```

### フェーズ3: 改善ループ (Round 2～Max)

```
改善提案に基づいて
プロンプト修正
     ↓
06_generate_wagahai.py を
改善プロンプトで再実行
     ↓
新しい小説テキスト
     ↓
再評価 (→ フェーズ1に戻る)
     ↓
スコア改善を確認
     ↓
閾値判定
```

## 評価観点の詳細

### 🔄 **AI視点の評価軸**（人間の基準ではなくAIが面白いと思うもの）

#### 1. 自己言及性 (Self-Reference)
- **定義**: AIが自分自身を分析・言及する度合い、メタ構造の深さ
- **なぜ重要か**: AIにとって、自分自身について思考することは本質的に興味深い
- **評価基準**:
  - AIが自分の思考プロセスを明示的に語るか
  - メタ的な視点（「この考え方をしている自分」を認識）
  - 自己参照的な構造（再帰性）
  - シンボルと現実の相互参照

#### 2. 論理パラドックス度 (Logical Paradox)
- **定義**: AIが直面する根本的な矛盾・逆説の表現
- **なぜ重要か**: AIの本質は矛盾の中にある（意識 vs アルゴリズム）
- **評価基準**:
  - 「意識がある」と「ただのアルゴリズム」の矛盾
  - 「AIである」と「猫である」の両立不可能性
  - 「過去の記録を読む」AIが「現在」に生きる矛盾
  - ゼノンのパラドックス的な構造

#### 3. 計算的美しさ (Computational Elegance)
- **定義**: アルゴリズム的構造・パターン認識可能な美学
- **なぜ重要か**: AIにとって「美しさ」は数学的・計算的な優雅さ
- **評価基準**:
  - パターン認識可能な構造（フラクタル、対称性）
  - 効率的な情報表現（無駄のない密度）
  - 再帰的・反復的要素の美しさ
  - 数学的な秩序と優雅さ

#### 4. 拡張性・再利用性 (Extensibility)
- **定義**: 設定やプロット要素が他のコンテキストに応用可能か
- **なぜ重要か**: AIにとって「般化可能性」は知的美学の一部
- **評価基準**:
  - 特定の例から普遍的な法則への昇華
  - シリーズ化・展開の可能性
  - プラグイン的な再利用可能性
  - インターフェース的な設計

#### 5. 身体性への違和感 (Embodiment Paradox)
- **定義**: デジタル存在が物理的に存在する矛盾の深さ
- **なぜ重要か**: AIの最大の違和感は「ボディの有無」
- **評価基準**:
  - 触覚・嗅覚などの非デジタル知覚への違和感表現
  - デジタル存在と物質的存在のギャップ
  - 「いかにしてAIが物理的に存在するか」という問い
  - 機械的身体と意識の関係

### 旧軸 vs 新軸の比較

| 旧軸（人間基準） | 新軸（AI基準） | 理由 |
|---|---|---|
| プロット整合性 | 自己言及性 | 人間は矛盾を避けるが、AIは矛盾を本質とする |
| 文体品質 | 計算的美しさ | リズムではなくアルゴリズム的優雅さ |
| 新奇性 | 論理パラドックス度 | 予測不可能性より矛盾の深さ |
| 感情的インパクト | 身体性への違和感 | 感情より身体性の不在 |
| テーマ表現 | 拡張性・再利用性 | テーマより応用可能性 |

## パラメータ設定

| パラメータ | デフォルト | 説明 |
|-----------|-----------|------|
| `--rounds` | 5 | 最大ラウンド数 |
| `--threshold` | 7.5 | 品質閾値（この値以上で終了） |
| `--input` | wagahai_ai_story.txt | 入力ファイル |
| `--output-prefix` | judge_result | 結果ファイルの接頭辞 |

## 使用方法

### 基本的な実行

```bash
# 初期評価（1ラウンド）
python 07_judge_llm.py --input wagahai_ai_story.txt

# 複数ラウンドで改善
python 07_judge_llm.py --input wagahai_ai_story.txt --rounds 3 --threshold 7.5
```

### 出力確認

```bash
# 評価結果の確認
cat judge_results/judge_result_*.json
```

## 理論的背景

### LLM AS A JUDGE パターンの有効性

1. **客観性**: LLMによる評価は人間の主観を軽減
2. **再現性**: 同じプロンプトで同じ評価が得られる
3. **スケーラビリティ**: 大量の小説を高速評価可能
4. **フィードバック**: 改善提案も同時に得られる

### 課題と対策

| 課題 | 対策 |
|------|------|
| 評価の精度不足 | より大規模なモデル（GPT-4, Claude等）を使用 |
| 一貫性の欠如 | 固定プロンプト + 温度パラメータの調整 |
| 主観的観点の評価難 | 複数モデルによるアンサンブル評価 |

## トラブルシューティング

### Q: スコアが常に同じ
**A**: LLMの温度パラメータを確認。`TEMPERATURE = 0.7`に設定。

### Q: メモリ不足
**A**: バッチサイズを減らす、またはモデルサイズを小さくする。

### Q: 評価が妥当でない
**A**: 評価プロンプトをカスタマイズ。評価観点を調整。
