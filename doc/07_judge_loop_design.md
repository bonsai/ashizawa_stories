# LLM AS A JUDGE - ループ設計ドキュメント

## 概要

「LLM AS A JUDGE」パターンは、別のLLMを評価者として用いて、生成された小説の品質を客観的に評価し、改善ループを自動化するアプローチです。

## アーキテクチャ

```
┌─────────────────────────────────────────────────────────┐
│                  07_judge_llm.py                        │
│  LLM AS A JUDGE パターンの実装                          │
└─────────────────────────────────────────────────────────┘
              ↑ 入力                     ↓ 出力
              │                         │
     ┌────────────────┐        ┌────────────────────┐
     │ 06_generate    │        │ judge_results/     │
     │ _wagahai.py    │        │ (評価ログ JSON)    │
     │ の出力ファイル  │        │                   │
     └────────────────┘        └────────────────────┘
```

## ループの流れ

### フェーズ1: 初期評価 (Round 1)

```
初期生成ファイル
     ↓
┌─────────────────────────────────────┐
│  LLMJudge.evaluate_story()          │
│  ・プロット整合性 (0-10)            │
│  ・文体品質 (0-10)                  │
│  ・新奇性 (0-10)                    │
│  ・感情的インパクト (0-10)          │
│  ・テーマ表現 (0-10)                │
│  ↓ 計算                             │
│  総合評価 = (各項目の平均)           │
└─────────────────────────────────────┘
     ↓
  スコア 出力
```

### フェーズ2: 判定と分岐

```
総合スコア >= 閾値 (デフォルト 7.5)
     ↙ YES           ↘ NO
   終了          改善提案生成
                     ↓
```

### フェーズ3: 改善ループ (Round 2～Max)

```
改善提案に基づいて
プロンプト修正
     ↓
06_generate_wagahai.py を
改善プロンプトで再実行
     ↓
新しい小説テキスト
     ↓
再評価 (→ フェーズ1に戻る)
     ↓
スコア改善を確認
     ↓
閾値判定
```

## 評価観点の詳細

### 1. プロット整合性 (Plot Consistency)
- **定義**: 設定された物語構成との整合性
- **評価基準**:
  - AI（猫）視点が一貫しているか
  - 2124年の時間設定に矛盾がないか
  - 「chat_history（古文書）」というモチーフが活かされているか
  - キャラクター（人間を下に見るAI）の一貫性

### 2. 文体品質 (Style Quality)
- **定義**: 日本語表現の美しさとリズム感
- **評価基準**:
  - 音韻的美しさ・リズムの良さ
  - 比喩表現の洗練度
  - 「吾輩はAIである。猫である。」の出だしの効果的な活用
  - 文の長短による表現のバリエーション

### 3. 新奇性 (Novelty)
- **定義**: オリジナリティと意外性
- **評価基準**:
  - AI視点の独特性（人間的ではない視点表現）
  - 予測不可能な展開
  - 既存の小説・文学との差別化
  - ナンセンス・ユーモアの活用度

### 4. 感情的インパクト (Emotional Impact)
- **定義**: 読み手への訴求力
- **評価基準**:
  - ユーモア・風刺の効果
  - 思想的な深さからの共鳴
  - 人間性への問い
  - 哀れみや皮肉の表現度

### 5. テーマ表現 (Theme Expression)
- **定義**: 中心的なテーマの表現度
- **評価基準**:
  - 「100年後の人間は何をしているか」というテーマ
  - AIの哲学的視点の深さ
  - 人間性 vs AI性の対比
  - 機械学習と人間知性の関係への洞察

## パラメータ設定

| パラメータ | デフォルト | 説明 |
|-----------|-----------|------|
| `--rounds` | 5 | 最大ラウンド数 |
| `--threshold` | 7.5 | 品質閾値（この値以上で終了） |
| `--input` | wagahai_ai_story.txt | 入力ファイル |
| `--output-prefix` | judge_result | 結果ファイルの接頭辞 |

## 使用方法

### 基本的な実行

```bash
# 初期評価（1ラウンド）
python 07_judge_llm.py --input wagahai_ai_story.txt

# 複数ラウンドで改善
python 07_judge_llm.py --input wagahai_ai_story.txt --rounds 3 --threshold 7.5
```

### 出力確認

```bash
# 評価結果の確認
cat judge_results/judge_result_*.json
```

## 理論的背景

### LLM AS A JUDGE パターンの有効性

1. **客観性**: LLMによる評価は人間の主観を軽減
2. **再現性**: 同じプロンプトで同じ評価が得られる
3. **スケーラビリティ**: 大量の小説を高速評価可能
4. **フィードバック**: 改善提案も同時に得られる

### 課題と対策

| 課題 | 対策 |
|------|------|
| 評価の精度不足 | より大規模なモデル（GPT-4, Claude等）を使用 |
| 一貫性の欠如 | 固定プロンプト + 温度パラメータの調整 |
| 主観的観点の評価難 | 複数モデルによるアンサンブル評価 |

## トラブルシューティング

### Q: スコアが常に同じ
**A**: LLMの温度パラメータを確認。`TEMPERATURE = 0.7`に設定。

### Q: メモリ不足
**A**: バッチサイズを減らす、またはモデルサイズを小さくする。

### Q: 評価が妥当でない
**A**: 評価プロンプトをカスタマイズ。評価観点を調整。
